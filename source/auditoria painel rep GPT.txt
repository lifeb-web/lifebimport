ITEM 1 — Token único exposto / sem autenticação real
STATUS: CORRIGIDO PARCIALMENTE
O que foi feito: Criados tokens individuais por rep (IRAMAR e NATANAEL com tokens distintos). O proxy agora valida token + rep juntos tanto no GET quanto no POST. Um token vazado de um rep não dá acesso ao outro.
O que ainda falta: Não há autenticação por sessão (login/senha). Para o perfil de uso atual (time interno, links privados com noindex), o nível de proteção é suficiente. Para escalar com reps externos, seria necessário OAuth ou JWT.
Ação necessária: Reimplantar o proxy no Apps Script (nova versão).

ITEM 2 — Endpoints retornam dados de todos os reps
STATUS: CORRIGIDO
O que foi feito: getActive(rep) e getRepHistory(rep) agora filtram no servidor antes de devolver qualquer dado. O JSON retornado contém SOMENTE os leads do rep autenticado. O filtro client-side foi removido do frontend.
Filtro de segurança adicional: Mantido um filtro client-side como fallback durante a transição (enquanto o proxy antigo ainda estiver ativo). Após reimplantação do proxy, esse filtro é um no-op (filtra uma lista já filtrada).

ITEM 3 — Limite de 200 global antes do filtro por rep
STATUS: CORRIGIDO
O que foi feito: O limite de 200 agora é aplicado APÓS o filtro por rep. Cada rep pode ter até 200 leads ativos e 200 no histórico, independentemente dos outros reps.

ITEM 4 — Atualizações não atômicas (múltiplos POSTs)
STATUS: CORRIGIDO
O que foi feito: Fechado e Perdido agora usam action='update_lead' — um único POST com LockService
no proxy. Todos os campos (status, valor, data, motivo, pot) gravados atomicamente.
Rollback visual: se o POST falhar, _revertOptimisticClose() restaura o lead na fila, remove
do histórico otimista e reverte o timestamp. Rep vê o lead de volta com estado original.

ITEM 5 — Fire-and-forget em algumas escritas
STATUS: CORRIGIDO PARCIALMENTE
Fechado/Perdido: VALOR_POTENCIAL=0 agora entra no mesmo update_lead atômico — não é mais fire-and-forget.
Residual aceito por design: valor potencial em "Em negociação" e reversões/undo ainda são fire-and-forget.
Para undo (ação rara e secundária), esse comportamento é aceitável.

ITEM 6 — Média de dias até fechar incorreta
STATUS: NÃO CORRIGIDO — não necessário
Motivo informado pelo dono do sistema: a planilha já tem fórmula que calcula DIAS_ATE_FECHAR automaticamente quando DATA_FECH é preenchida. O CRM salva DATA_FECH corretamente. Não há necessidade de calcular no proxy.

ITEM 7 — Múltiplas fontes de verdade no template
STATUS: CORRIGIDO

CORRECOES E MELHORIAS DESTA SESSAO

1. BUGFIX: historico nao abria aba Fechados/Perdidos apos clicar
   Causa: setHistTab() substitui body.innerHTML removendo o botao do DOM.
   O evento de click continuava bubblando ate main; e.target.closest() retornava null
   porque o elemento nao existia mais. collapseHistorico() era chamado imediatamente.
   Correcao: guard document.contains(e.target) — se o alvo foi removido do DOM
   pelo proprio onclick, o listener de main ignora o evento.

2. Logo clicavel = refresh da pagina
   onclick="location.reload()" + cursor:pointer no img.h-logo.
   Remove filtros e forca carregamento fresco sem indicador visual adicional.

3. Cache de leads isolado por rep (impede contaminacao cruzada)
   Causa do bug: ambas as paginas rodam no mesmo dominio (projetojlbv.com.br) e
   compartilham localStorage. Cache antigo ou variavel JS podia vazar dados de
   Iramar para Natanael na carga rapida.
   Correcao dupla:
   a) CACHE_KEY agora e string literal 'lifeb_cache___REP_NAME__' substituida pelo
      Python em build time — nunca depende de variavel JS em runtime.
   b) Cache salvo com { rep: REP_NAME, leads, ts }. Ao ler, valida cached.rep === REP_NAME.
      Se nao bater, remove a entrada e mostra skeletons.

4. Fechamento em dois passos separados (UX)
   Problema: valor e data apareciam juntos na mesma linha flex, quebrando o card no mobile.
   Solucao: fluxo sequencial controlado por prompt.dataset.step:
   - Passo 'valor': exibe somente o campo de valor (R$ com preview em tempo real).
     Botao "Avancar ->". Enter avanca para o proximo passo.
   - Passo 'data': esconde valor, exibe somente o campo de data (pre-preenchido com hoje).
     Botao "Salvar". Enter salva.
   O valor do passo 1 e guardado em prompt.dataset.valorFech.
   O bloco 'fechado' em saveStatusExtra foi extraido para bloco proprio com early returns,
   separado completamente do bloco 'perdido'.
   CSS: .lc-sp-row agora e display:block; .lc-sp-date-wrap e .lc-sp-pfx-wrap tem width:100%.

PROBLEMAS TRATADOS NESTA SESSAO

1. Timestamp zerado ao trocar de dispositivo
   Problema: _tsMap vive no localStorage do dispositivo. Se o rep abre o painel no celular
   e depois no computador, nao via os horarios de ultima atualizacao dos outros.
   Solucao implementada:
   - Nova coluna TIMESTAMP_VEND (AF = indice 31) na planilha — grava epoch ms a cada update
   - getActive retorna ts_vend: Number(r[31]) || 0 por lead
   - loadData mescla ts_vend do servidor em _tsMap (prefere o mais recente entre local e servidor)
   - Resultado: rep nunca perde o horario de ultima atualizacao, independente de dispositivo

2. Risco de fechamento/perda parcial (escrita nao atomica)
   Problema: Fechado fazia 4 POSTs sequenciais; Perdido fazia 2-3 POSTs.
   Em celular fraco ou internet instavel, poderia falhar no meio e gerar lead em estado inconsistente.
   Solucao implementada:
   - Nova action update_lead no proxy — usa LockService.getScriptLock()
   - Recebe campo { token, rep, row, action:'update_lead', updates:{...} }
   - Grava todos os campos em sequencia sob lock + escreve TIMESTAMP_VEND = Date.now()
   - Retorna { ok: true, ts: <epoch_ms> }
   - Frontend atualiza _tsMap[row] com ts do servidor apos resposta bem-sucedida
   - Fechado: 4 POSTs → 1 POST atomico
   - Perdido: 2-3 POSTs → 1 POST atomico
   - Fire-and-forget do VALOR_POTENCIAL=0: resolvido — entra no mesmo update_lead

3. Timestamp fallback para leads sem historico (commit 3d5c89ff)

IMPORTANTE — Por que Date.now() e nao new Date().toISOString()

O Google Sheets tem comportamento de auto-conversao: se uma celula recebe uma string no
formato ISO 8601, o Sheets pode converte-la para um objeto Date internamente. Ao ler de
volta via getValues(), seria retornado um Date do GAS, e quando norm(v) fosse aplicado
retornaria uma string como "Sat Apr 26 2026 01:23:45 GMT+0000" — nao parseable pelo
frontend com new Date(str).

Date.now() retorna um numero inteiro (ex: 1745648025000). Sheets armazena como numero.
getValues() retorna o mesmo numero. Number(v) no frontend funciona diretamente.

PROBLEMA
O doPost da planilha de leads (script que recebe submissions do site) tinha tres riscos:
1. appendRow + setValues em chamadas separadas sem LockService — race condition possivel
   se dois leads chegassem ao mesmo tempo, as UTMs podiam ir para a linha errada.
2. Sem try/catch — JSON malformado ou qualquer erro causava crash sem resposta.
3. Sem validacao — POST com nome e telefone vazios inseria linha em branco na planilha.

CORRECAO APLICADA
- LockService.getScriptLock() envolve todo o doPost — appendRow + setValues atomicos
- try/catch retorna { status:'erro', error:'...' } em vez de crashar
- Validacao: nome e telefone obrigatorios — retorna erro sem inserir nada se vazios

TESTE REALIZADO
- POST com nome e telefone vazios → { "status":"erro","error":"nome e telefone obrigatórios" }
- POST valido → { "status":"ok" } + linha inserida corretamente na planilha

Duplicidade: nao tratada por codigo — SDR identifica e marca status "Duplicado" manualmente.

onEdit nao foi alterado — sem necessidade.

BUG 1 — Botão de contato ficava roxo preso (commit 872b209f) — CORRIGIDO
  Problema: ao clicar em WA ou Maps no card, o painel de contato fechava mas o
  botão de contatos (ícone pessoa) permanecia visualmente ativo (roxo) incorretamente.
  Causa: openLinkConfirm() buscava `.lc-icon-btn.info` para remover o estado active,
  mas a classe real do botão é `.lc-icon-btn.contact`. querySelector retornava null,
  o classList.remove('active') nunca executava.
  Fix: `.lc-icon-btn.info` → `.lc-icon-btn.contact` (linha 999 do template).

BUG 2 — Div #fb-{row} morta em cada card (commit f7fb07ef) — CORRIGIDO
  Problema: cada card renderizava um <div class="lc-feedback" id="fb-{row}"></div>
  que nunca era referenciado por nenhum getElementById no JavaScript.
  O feedback de obs usa obs-fb-{row} (correto). O fb-{row} era dead code de versão anterior.
  Fix: div removida. CSS .lc-feedback preservado (ainda usado pelo obs-fb-).

BUG 3 — WA contato empresa sem WA principal sumia silenciosamente (commit 643c9d1f) — CORRIGIDO
  Problema: overlay lcp-wac-{row} só era renderizado dentro de ${wa ? ... : ''}.
  Se lead não tinha l.whatsapp mas tinha l.contato com WA, o botão de contato empresa
  existia, mas ao clicar openContactWa() buscava getElementById('lcp-wac-'+row) = null
  e retornava silenciosamente — nada acontecia, sem feedback ao rep.
  Fix: condição trocada de `wa` para `hasContact`. Overlay existe sempre que o painel
  de contato existe. Mesma condição já usada para o botão de contatos (linha 1749).
